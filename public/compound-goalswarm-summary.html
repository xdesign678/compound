这次任务为什么重要

Compound 是用户的个人知识库，每天高频使用。如果用户在手机上浏览概念列表、点开详情阅读后返回，页面却回到最顶部，就会打断阅读节奏、增加重复滚动成本。类似的，如果下拉刷新和边缘返回手势互相抢触摸事件，或者 Tab 点击没有即时的「按下去」感觉，都会让产品显得「不跟手」。这些细节看似小，但每天重复几十次就会累积成「这个产品有点卡」的整体印象。

发现了什么 或 修了什么

P0 — 阻断级（已修复）

Wiki 模式滚动位置丢失

修复前，用户在首页 Wiki 模式向下滚动约 400px，点开任意概念卡片进入详情覆盖层，再点击返回后，.app-main.scrollTop 从 400 归零到 0，100% 复现。

根本原因是 LibraryView 和 SourcesView 已有 scrollTop 保存/恢复机制，但 默认首页的 WikiView 完全缺失。git log 中两次声称「彻底解决」的 commit 实际只修复了 Library 模式，未覆盖 Feed/Wiki 模式。

修复方案： - 在 lib/store/ui-slice.ts 和 lib/store.ts 中新增 wikiState，记录 scrollTop 与可见概念锚点。 - 在 components/views/WikiView.tsx 中补充滚动保存与恢复逻辑，并为每张 Wiki 卡片打上 data-concept-id，让恢复时可按稳定概念 id 定位。 - 优化主滚动容器为 overscroll-behavior-y: contain，限制纵向 overscroll，避免误触发系统级刷新。

修复后 Playwright 实测：before=420、after=504、preserved=true。

后续建议（非本次阻断）

  1. viewport maximumScale: 5 + userScalable: true 与 PWA standalone 体验冲突：当前设置允许用户双指缩放，破坏「原生 App」错觉。本次未修改，建议后续根据 display-mode: standalone 动态注入 viewport。
  2. 移动端详情覆盖层存在两个「返回」按钮：截图与 accessibility tree 均确认有两个 button "返回" 同时存在，可能导致误触。本次未修改，建议后续在移动端 overlay 模式下隐藏 ConceptDetail 内嵌的返回按钮。
  3. 生产 Service Worker 离线能力未实测：开发模式下 ServiceWorkerRegister 会主动 unregister SW，这是设计如此。但真实离线 fallback 行为（断网后访问 / 能否自动导航到 /offline）只能在生产构建后验证，本地 dev 环境无法模拟，标记为 blocked。

P2 — 中优先级（已修复）

  1. 底部 Tab 点击无 :active 背景色反馈：原有 .tab-item:active 只有 transform: scale(0.96) 微缩动画，没有背景色变化。已补充短暂背景色反馈，让触摸点击有明确的「按下」视觉。
  2. PullToRefresh 与 SwipeBack 左边缘冲突：两组件各自硬编码边缘阈值（36px vs 20px），容易在调整时遗忘同步。已将边缘常量抽取到 lib/gesture-edges.ts，两组件共用同一阈值，并新增单元测试 lib/gesture-edges.test.ts 覆盖。

P3 — 低优先级（建议后续优化）

z-index 硬编码未完全统一为 CSS 变量：reviewer 确认 .mobile-detail-overlay 已正确切换为 --z-modal（60），与 CommandPalette（200）、Toast（80）、TaskCenter（70）、SwipeBack 指示器（100）层级关系正确无冲突。但 SwipeBack 指示器的 z-index: 100 仍是硬编码，建议后续统一迁移为 CSS 变量。
