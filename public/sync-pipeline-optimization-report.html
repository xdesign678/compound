两份意见的核心并不冲突。合并后的判断是：流程决定了排队时长和错误体感，模型决定了单次调用下限，GitHub 拉取本身很快。

60%

同步流程

单车道、增强阶段阻塞首屏、延迟任务缺少自动唤醒，以及前后端超时口径不一致。

90 天诊断样本中单次调用平均约 18 秒，概念摄入平均约 30 秒；输出截断会造成重试。

1107 个文件的目录扫描诊断约 2.2 秒；它不是几小时队列和 finish_length 的来源。

诊断基线，不是优化后的成绩： 历史样本包含 4038 次 LLM 调用；曾出现 251 个文件约 2 小时、929 个文件约 13.4 小时的运行。 这些数据用于定位旧链路瓶颈，优化后的真实 p50 / p95 需要在新版本部署后重新采样，报告不做虚假提速承诺。

Architecture

架构从串行等待变成分层交付

最重要的变化不是“把并发数字调大”，而是重新定义什么叫完成：资料和概念写入后即可使用，增强分析不再绑架核心结果。

优化前 · 一条长链
1GitHub 拉取文件文件进入统一队列。
2单并发 LLM 摄入大量文件排成一列。
3同步等待情境与关系增强任务拖住用户可见结果。
4整批结束后刷新中间产物已经存在，前端却看不到。
5失败显示为整文件失败甚至可能只是非关键增强失败。
优化后 · 核心优先
1Webhook / 定时 / 手动触发只扫描真实变化，GitHub 继续做源文件库。
2核心摄入安全并发 2资料、概念、分块先写入 SQLite。
3完成一个就同步一个浏览器按变更游标拉增量快照。
4增强任务分级排队摘要 50、向量 40、关系 15、情境 5。
5失败可降级、重试自唤醒核心内容仍可用，问题中心保留告警。

Implementation

具体优化了什么

改动覆盖请求时限、队列、模型输出、调度、前端反馈和运维可观测性。每一项都围绕“更早看到结果、少报假错、失败后能恢复”。

01 · 首次结果

把情境索引移出核心摄入

资料、概念与基础分块提交后立即返回；耗时的情境前缀改由低优先级后台任务完成，不再阻塞用户看到总结。

02 · 并发策略

Operational notes

仍需注意什么

代码层已经完成，但有两类结果不能靠本地测试伪造：外部平台配置是否存在，以及新版本在真实负载下的 p50 / p95。

上线后优先核对

GitHub Webhook 是外部配置

仓库端点、签名校验和诊断都已就绪；仍需在 Zeabur 配置 GITHUB_WEBHOOK_SECRET，并在 GitHub 仓库确认 push 投递成功。

持续观察

并发采用保守默认值

核心并发从 1 提到 2，而不是直接设为 4。若 429、SQLite busy 和失败率稳定，再按阶段逐级提高。

数据说明

历史耗时不可当作新成绩

旧运行数据用于根因判断。部署后建议采集至少一轮 100+ 文件同步，再比较首个结果时间、总耗时与失败率。

上线观察项建议口径判断标准动作
首个文件可见时间从触发同步到浏览器出现首条新资料应显著早于整批完成若仍等待整批，检查增量快照游标
LLM 等待与 429按阶段观察队列等待、429 和网关超时吞吐提升且错误率不抬升只调整对应阶段并发
模型输出失败finish_length、reasoning_only重试后大部分恢复持续失败则替换 JSON 更稳定的模型
Webhook 投递配置状态与 processed 记录push 后出现增量运行检查 secret、Payload URL 与 GitHub Delivery

Decision

关于 GitHub 的最终决定

不迁移。GitHub 继续负责版本化原文，SQLite 继续负责运行状态、概念、摘要和索引。问题出在处理流水线，不在文件存放位置。

保留 GitHub 的版本历史、Obsidian 兼容性和审计能力；通过 Webhook + 增量 Compare 缩短触发路径，而不是换一个存储再重写整套同步逻辑。
