AI 长期记忆方案研究：Compound 不该照搬，而该组合吸收

这份研究把 GBrain、Microsoft GraphRAG、Graphify 和 LLM Wiki 放在一起比较，结论不是换成某个开源框架，而是明确 Compound 自己该走的路线：用可读 Wiki 做产品骨架，用轻量图谱增强检索，用 SQLite 和审查机制保证可追溯。

一句话结论

Compound 最适合做成“私有、可审查、可导出的 LLM Wiki”。 它不需要把系统替换成 GBrain、GraphRAG 或 Graphify，而应该把这些方案里最适合自己的部分拆出来：LLM Wiki 负责产品形态，GraphRAG-lite 负责关系检索，GBrain 负责区分知识记忆和运行状态，Graphify 只作为开发维护辅助。

LLM Wiki：产品模型 GraphRAG-lite：检索模型 GBrain：运营记忆边界 Graphify：代码地图辅助

为什么重要

避免架构误判

不是看到明星项目就迁移

星标数只能说明关注度，不能证明适合 Compound。研究明确指出：外部项目应该作为参考，而不是成为新依赖或重写理由。

保护核心资产

知识必须可读、可追溯

Compound 的价值不只是“能回答问题”，而是把原始材料、概念页、证据、关系和版本历史连起来，让人可以审查和导出。

控制成本

轻量图谱优先

完整 GraphRAG 对个人或小团队知识库偏重。更稳的路线是先做 typed relations、证据片段、主题摘要和低置信度审查。

服务现有系统

把已有 SQLite 基础用好

仓库已经有概念、来源、证据、关系、审查队列和模型运行记录。升级重点是闭环一致性，而不是另起炉灶。

参考项目

研究对照了三个 GitHub 开源项目和一个知识库设计模式。下面的“借鉴点”是适合 Compound 吸收的部分，不等于要直接引入为运行时依赖。

参考 它解决什么 Compound 应该怎么用
GBrain 面向运行中 AI agent 的长期大脑，强调持久知识和运行状态的分离。 借鉴“知识记忆”和“运营记忆”分层，不作为产品运行时依赖。
Microsoft GraphRAG 从非结构化文档中抽取实体、关系和社区摘要，增强跨文档综合问答。 先做 GraphRAG-lite：概念关系、证据、置信度、审查门禁和主题摘要。
Graphify 把代码项目变成可查询知识图谱，帮助 coding agent 理解仓库结构。 只用于生成本地代码地图，帮助维护者和 agent 定位模块，不进入生产路径。
LLM Wiki 一种用 LLM 维护 Markdown 知识库的产品模式，强调原始材料和编译 Wiki 分层。 作为 Compound 的核心产品模型：人能读、Git 能版本化、系统能追溯来源。

适配判断

Compound 的主线不是“换框架”，而是做一个混合体：私有 LLM Wiki + SQLite 证据链 + 轻量图谱 + 审查队列 + 可导出的 Markdown。

已有基础

sources 保存原始材料，concepts 保存 Wiki 页面，concept_evidence 连接证据，concept_relations 是图谱层，Ask 检索已经有改写、混合召回、图扩展、重排和引用检查。

主要缺口

研究认为缺口不是大架构，而是闭环：关系抽取要真正落地，概念改动要同步索引和版本，Wiki 搜索要使用服务端检索能力，Markdown 导入导出要支持 frontmatter 和 roundtrip。

升级方向

1

数据一致性优先

所有概念改动都要同步更新概念表、版本表、全文索引、证据、关系和活动记录，避免 UI、Ask 检索和导出结果互相漂移。

2

GraphRAG-lite 关系层

抽取 supports、depends_on、contradicts、same_as 等关系，并为每条边保留置信度、解释和证据。

3

审查门禁

低置信度关系不能悄悄改图。它们应该进入 review queue，让人选择批准、拒绝或只标记已处理。

4

Markdown 往返

导出 wiki/index.md、wiki/concepts/*.md、wiki/graph.json 和主题摘要；导入时按稳定 ID 回写版本并重建搜索与图谱。

5

检索和运营记忆升级

UI 搜索使用服务端 Wiki context API；模型用量、成本、延迟、失败、同步和分析积压作为运营记忆展示出来。

风险和注意事项

不要用星标数做决策

stars 是热度信号，不是架构适配度。真正要看的是 Compound 的数据模型、成本、可审查性和用户工作流。

不要每次小改都跑完整 GraphRAG

完整索引成本高，容易慢。应先做增量关系、局部主题摘要和重要来源回填。

不要让 Markdown 脱离 SQLite

除非设计完整 canonical-file 模式，否则 SQLite 仍应是可追溯事实层，Markdown 是可读与可导出的表现层。

不要丢掉 provenance

LLM Wiki 会压缩信息，压缩本身有损。每个结论都要能回到来源、证据和版本。

下一步

1

补评测

为关系抽取和图扩展增加 evaluation fixtures，避免图谱增强后出现“看起来更聪明、实际更不准”。

2

补可视化

增加单个概念的一跳、二跳关系图，让用户能看见系统为什么把两个知识点连起来。

3

补导入预览

Markdown roundtrip 不应只有 API。导入前要让用户预览会新增、更新、冲突哪些概念。

4

补成本和回填

当 model_runs.cost_usd 数据足够后加按日成本图；对重要来源执行重建索引和关系抽取回填。
