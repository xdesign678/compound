2. 这次任务为什么重要

Compound 是一个个人知识管理工具（PWA），用户既会在电脑上整理资料，也会在手机上随时复习和查看。如果同一网址在手机和电脑上显示完全不同的功能，或者菜单点不透、到处飘红报错，用户会立刻觉得这个产品"不靠谱"，直接关闭页面。

这次走查覆盖的是 线上真实部署环境，不是本地开发机，所以发现的问题都是真实用户正在遇到的。

5. 发现了什么 / 修了什么

🔴 P0 — 阻断级（1 个）

P0-001: /review 同一网址在电脑和手机上显示完全不同的功能

  • 来源: T4-ui-review-recap
  • 证据: agents/T4-ui-review-recap-report.md、evidence/review-desktop.png、evidence/review-mobile.png
  • 现象：
  • 电脑端打开 https://compund.zeabur.app/review → 看到"概念复盘队列"，有 9 个概念卡片，可以按分类筛选。
  • 手机端打开同一个网址 → 看到"审核队列"，内容变成"批量入库、低置信度抽取、同步冲突"，还有一大块红色"需要认证 Unauthorized"报错。
  • 严重程度：这不是"电脑版功能多、手机版精简"的正常响应式差异，而是完全两个不同的功能塞进同一个网址。用户收藏了网址在电脑上复习，换手机打开却变成一个审核后台——会以为网站被黑了。
  • 建议：两个功能必须用不同的网址（比如 /review 和 /audit）；如果未登录，手机端也应该显示同样的复习内容（只读模式），而不是跳到一个毫不相关的页面。

对你的意义：你发给朋友的复习链接，朋友用手机打开后看到的完全是另一套东西，直接懵掉。

🟠 P1 — 高优先级（5 个）

P1-001: 未登录时控制台疯狂报 401 错误

  • 来源: T2/T3/T4/T5 均报告
  • 证据: 各页面 console 日志截图
  • 现象：不登录访问任何页面，浏览器控制台会刷出 6-10 条红色 401 Unauthorized 错误，内容是 [cloud-sync] snapshot pull failed。
  • 影响：虽然普通用户不会天天开着控制台，但开发者看到的全是噪音；更关键的是这说明未登录体验没有被优雅处理——系统在不断尝试做一件没权限做的事，而不是安静跳过。
  • 建议：未登录时直接跳过云同步拉取，不要在控制台抛错；或在 UI 上给一个温和的"登录以开启同步"提示。

P1-002: 电脑三栏布局中间太挤、右边太空

  • 来源: T3-ui-library
  • 证据: evidence/desktop-sources-list-v3.png
  • 现象：在 1280-1440px 宽度的电脑上，库页面左边导航栏 + 中间资料列表 + 右边详情预览的三栏布局中，中间资料列表被压得很窄，卡片标题 truncation（显示不全）严重，而右边详情区留白过大。
  • 建议：把三栏比例从约 20%/35%/45% 调为 18%/42%/40%，或者在 1280px 以下直接改为两栏布局（左边收成小图标，中间列表占满）。

对你的意义：用笔记本看资料库时，中间列表挤得看不清标题，像在看一列被压扁的便利贴。

P1-003: 手机"更多操作"菜单点不透（click-through）

  • 来源: T5-ui-sync-global / T2-ui-home
  • 证据: evidence/mobile-more-menu-open.png、evidence/settings-mobile-final.png
  • 现象：手机端点击右上角"⋮"更多操作按钮，菜单展开后，再点菜单里的"设置"或"从 GitHub 同步"，菜单收起来了，但对应的弹窗/页面没打开，反而触发了菜单下面那个位置的按钮（比如某个分类 pill 或概念卡片），直接跳到了别的页面。
  • 严重程度：这是典型的移动端触摸事件穿透（click-through）bug，菜单没拦住手指的点击，让它漏到了下面的元素上。这是严重的交互缺陷。
  • 建议：给菜单容器加上事件拦截（stopPropagation / preventDefault），或者把菜单改成真正的模态浮层（Portal），确保菜单打开时下面的页面不可点击。

对你的意义：手机上的菜单点了跟没点一样，反而莫名其妙跳到了别的页面，很烦躁。

P1-004: 手机"问答"页面完全不可用，显示的是首页

  • 来源: T5-ui-sync-global（原评 P2，升级为 P1）
  • 证据: evidence/qa-mobile.png、evidence/qa-mobile-v2.png
  • 现象：手机端访问 /qa 或点底部 TabBar 的"问答"Tab，页面内容和首页一模一样，没有问答专属内容。同时 TabBar 里"问答"也没被高亮，高亮的还是"Wiki"。
  • 严重程度：整个问答功能在移动端完全缺失。
  • 建议：检查 /qa 路由的渲染逻辑，确认没有错误地 fallback 到首页组件；同时修复 TabBar 的当前选中状态。

对你的意义：手机上的"问答"入口是个摆设，点进去还是首页。

P1-005: 未登录状态的红色认证横幅太吓人

  • 来源: T4-ui-review-recap / T5-ui-sync-global
  • 证据: evidence/sync-desktop.png、evidence/review-mobile.png
  • 现象：未登录时访问 /sync 或 /review，页面顶部会出现大块粉红色/红色的报错卡片，写着"需要认证 Unauthorized"，还有一个"重试"按钮——但不登录的话，点多少次重试都没用。
  • 影响：大面积红色会制造恐慌感，让访客以为网站坏了。实际上未登录访客本来就应该看到这些功能的只读版本或引导登录。
  • 建议：把红色错误态降级为温和的蓝色/灰色信息卡片，文案改成"登录后查看同步状态"，去掉无效的"重试"按钮。

对你的意义：新访客打开页面就被一片红吓到，第一反应是"这个网站是不是挂了"，然后关掉走人。

10. 风险和注意事项

已升级但未解决的问题

  • reviewer 建议升级到 swarm-reviewer-opus（Opus 4.7 Max）对 P0-001 做深度安全/路由审计：因为 /review 在桌面端和移动端渲染完全不同的内容，这涉及中间件或路由守卫的分支逻辑， reviewer 担心背后可能有数据泄露或认证绕过的风险。本次未实际升级，需要 commander 决定是否追加专项审计。

遗漏的审计范围

  • 安全审计（高）：XSS、CSRF、CSP、认证 token 存储方式均未检查。
  • PWA / ServiceWorker 生命周期（中）：误离线行为暗示 ServiceWorker 缓存/更新逻辑可能有问题，但没有深入检查 sw.js 代码。
  • API 响应内容正确性（中）：smoke 测试只验证了 HTTP 200 和页面能打开，没验证返回的 JSON 数据是否正确。
  • 暗色模式（低）：未测试。
  • Lighthouse 性能量化（低）：只记录了 First Load JS 大小，没有跑 Lighthouse 性能分数。
  • 跨浏览器兼容性（低）：只在 Chromium 内核浏览器测试，未覆盖 Safari（iPhone 默认浏览器）和 Firefox。

环境依赖

  • 本次审计基于未登录（匿名）状态。登录后 cloud-sync 401 错误会消失，部分弹窗交互可能有不同表现。
  • better-sqlite3 模块版本不匹配只在本地开发环境出现，不影响线上 Zeabur 部署。
  • Smoke 测试响应时间受网络影响，首页 / 的 309ms 可能是冷启动或 SSR 开销，不是性能问题。

11. 下一步建议

  1. 立即修复 P0-001：这是唯一能让用户"以为网站坏了"的阻断 bug，优先级最高。
