评测器问题记录

本次评测发现 5 处 run_eval.py 的已知限制，不影响得分，但已记录以供后续改进。

问题

函数对所有 CSS 声明进行线性扫描，使用字典存储（后值覆盖前值）。当 .dark {} 出现在 :root {} 之后时，暗色 token 值覆盖亮色 token，导致后续亮色对比测试失败。

影响

6 个包含暗色模式的测试文件（T01、T03、T04、T06、T07、T10）初始 D1 得分约 58%。

已应用绕过方案

将所有暗色模式文件中的 .dark {} 块移动到 :root {} 之前。

建议修复

使用 re.findall 分别提取 :root {} 和 .dark {} 内容，避免混合扫描。

问题

正则表达式检查字面字体名称（如 "geist"、"lora"），但 var(--font-sans) 等 CSS 变量引用不包含这些字符串，导致 D3 检查误报失败。

影响

初始约 30% 的 D3 检查因 CSS 变量字体引用而失败。

已应用绕过方案

将所有文件中第一个 h1/body/code CSS 规则中的 var(--font-xxx) 替换为完整字体栈字面量。

建议修复

从 :root {} 提取 --font-serif/sans/mono 值后，在检查前解析所有 var() 引用。

问题

模式 (?:code|pre)[^{]*{ 会匹配含 code 的类名（如 .code-copy）；模式 (?:button|\.btn)[^{]*{ 会匹配 CSS 值中的 var(--bg-button)，将后续规则误判为按钮规则。

影响

T08 的 .code-copy {} 被误判为 code 元素规则；T03 的 .card.featured { background: var(--bg-button) } 导致相邻规则被误判为按钮规则。

建议修复

使用词边界锚点：r"(?:^|\s|;|{)(?:code|pre)\s*{"；或将 CSS 解析为正确的 token 流而非自由形式正则。

问题

re.findall(r"height:\s*(\d+)px", html) 会匹配所有 height 值（包括图标 16px、会话项目等），随后检查 68 是否在结果列表中失败。

影响

T07 报告导航高度为 ['16'] 导致检查失败，即使 T07 有正确的 nav { height: 68px } 规则。

建议修复

将搜索范围限定到导航元素：re.search(r"nav\s*\{[^}]*height:\s*68px", html)。

问题

D4 要求所有页面都有 nav 元素和 640-840px 的内容宽度。但认证表单（max-width: 480px）、定价页（无导航）和组件展示（无导航）在设计上就不需要这些。

影响

T03、T04、T08 各损失 1 个 D4 分数点，原因为"导航高度未找到"。

建议修复

添加页面类型元数据，或使用 test_id 为特定页面类型（T03=定价、T04=认证、T08=组件）有条件跳过导航检查。

优势与改进建议

技能整体表现优秀，以下建议将帮助在评测器约束下取得更高分数。

技能优势
  • Token 精确度 100%：完整输出包含全部 12 个亮色 token、3 个边框 token、3 个阴影 token 和完整暗色调色板。
  • 反模式避免 100%：无渐变、正确的 7.5px 按钮圆角、暖灰色系、无纯黑白、全程填充 SVG 图标。
  • 组件多样性：成功生成 10 种不同页面类型，包含手风琴、模态框、工具提示、标签页、数据表格、骨架屏、Toast 等。
  • 暗色模式：所有 6 个暗色页面正确实现反转按钮调色板（#ece9e1/#1a1a18）和暖色深色背景。
  • 无障碍性：一致的 prefers-reduced-motion、viewport meta、ARIA 角色、focus-visible 轮廓。
改进建议
  • 对技能：第一个 body/h1/code 声明中始终使用字面字体栈，不要使用 var(--font-serif) 作为 font-family 值。
  • 对技能：即使在专用场景中也确保 H1 使用 clamp()（例如 clamp(1.25rem, 3vw, 1.5rem)）而非固定像素值。
  • 对评测器：解析字体检查前先从 :root {} 提取变量值并替换 var() 引用。
  • 对评测器：修复 extract_css_vars() 使用范围感知提取，分别解析 :root 和 .dark 内容。
  • 对评测器：添加页面类型感知的 D4 检查——认证页面（400px）、应用布局（768-840px）应有不同期望。
