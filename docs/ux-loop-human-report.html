小白版解释：现在不是只靠“我看着还行”，而是有机器真的打开页面、点击功能、检查安装能力、检查无障碍、截图对比、模拟真实用户流程。

这套“体检机器”是什么

它像一个自动质检员

它会自动打开 Compound，进入不同页面，检查页面是不是能访问、按钮是不是有名字、离线页是不是存在、截图有没有明显跑版。

  • Playwright 负责像真人一样打开浏览器和点击页面。
  • Lighthouse 负责给 PWA、无障碍、最佳实践打分。
  • axe 负责专门检查严重无障碍问题。
  • sharp 负责对比新旧截图，防止页面跑版。

它不是全天自动监控

现在它主要是“手动触发，自动检查”。你可以让我运行，也可以以后接到 GitHub Actions，让每次推送后自动跑。

  • 查单个页面：npm run audit:ux -- --page=globalShell
  • 查全站 Lighthouse：npm run audit:ux:all
  • 查全站无障碍：npm run audit:ux:axe
  • 查真实流程：npm run test:e2e:stories

整体提升估算

这些百分比不是线上用户数据，而是按工程质量、PWA 完整度、无障碍、离线兜底和回归保护做的估算。已经有实测数据的地方会标出真实结果。

PWA 安装和离线完整度 +35% ~ +50%

补齐 manifest、图标、iOS 启动图、离线页、Service Worker 更新策略。

无障碍和键盘可用性 +20% ~ +30%

最终 axe 严重/高危问题为 0/0，Lighthouse 无障碍 100。

断网、慢网、失败兜底 +35% ~ +45%

离线页、离线重试、慢网问答、错误边界、同步失败反馈都做了补强。

以后防止改坏的能力 +70% ~ +90%

新增 21 个 surface 体检、42 张视觉基线、5 条用户故事和跨页扫描。

每一步做了什么

下面按真实任务卡顺序列出来。你可以把它理解成：先装体检设备，再补 App 基础能力，再逐页修体验，最后全站验收。

最重要的收益

短期收益

长期收益

  • 以后改功能前后可以跑体检，减少偷偷改坏的风险。
  • 全站关键页面有截图基线，可以发现明显跑版。
  • 关键用户路径有自动测试，不只靠人工点。
  • 质量标准写进项目里，新对话或新开发者也能照着执行。

还可以继续升级什么

现在是本地可手动触发的体检系统。下一步如果要更像成熟产品，可以把它接到自动化平台。

推荐下一步

  • 接入 GitHub Actions，让每次推送后自动跑核心体检。
